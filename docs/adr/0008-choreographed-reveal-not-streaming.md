# The frontend reveals the response in three choreographed acts, not as a stream

The browser UI receives the full `QueryResponse` as one JSON blob from `POST /query`, then reveals it in three timed acts on the client: the **routing plan**, then each **trace** step, then the composed answer. We rejected the obvious alternative of streaming chunks from the orchestrator because two of the three things we'd want to stream — the routing plan and the per-step trace outcomes — are interesting to stream, but the third — the `answer` string itself — is composed only after all tools finish (`handle-query.ts:201-232`), so it can only "stream" at the end anyway. That asymmetry undermines the real motivation for streaming, while preserving its costs: a chunked-transfer protocol, a per-step serialisation schema, and *uncontrolled* pacing — a happy MET day reveals everything in 600ms (no narrative), an unhappy day in 6s (visitor leaves). The cost of the choreographed approach: we cannot honestly claim per-step latencies (the trace schema does not carry them and we never render them); a pedantic engineer might ask whether the visual order matches the real execution order, to which the answer is "yes — we replay the trace in array order, which is the orchestrator's execution order." If we ever want true wall-clock-honest streaming, the orchestrator needs to be refactored to emit `plan`, `step`, and `answer` chunks, and the frontend needs to read them — at which point the choreography layer becomes vestigial and should be removed, not retained.
